Languages Supported

Detect vs. translate — two different numbers

Detection and translation are separate stages, so the counts differ:

  • Detects (spoken language — automatic): ~99 languages. You never tell the app what language is being spoken. It auto-detects the speaker's language on every turn, so it recognises essentially the full range of languages the speech engine understands.
  • Translates into (the "I want to hear in" dropdown): 37 target options. 31 languages, plus English in 6 variants (5 regional dialects and a Clear / Accent-Free mode). You pick one target; everything spoken in any other language is translated into it.

In practice: someone can speak in almost any language, and you hear it in whichever of the 24 target languages you chose.

English dialects & the Clear / Accent-Free mode

English is offered in regional dialects — US, UK, India, Australia, Canada — so translations come out in the spelling and idiom you expect.

English (Clear / Accent-Free) is a dedicated mode for international meetings: it takes English spoken in any accent (and any other language) and re-renders it as clean, neutral, standard English — both on screen and as audio. Unlike every other target, it never stays silent on English input, so a heavy accent or broken grammar is smoothed into clear English everyone can follow.

Same-language passthrough

If someone is already speaking your chosen language, no translation is produced — that turn is marked "already in your language — no translation needed" and stays silent. The one exception is English (Clear / Accent-Free), which always produces clarified output even for English input.

More target languages can be added easily — the underlying model already supports far more than 24; the dropdown list is simply the curated set enabled for this build.

Architecture & Security — how Orka Translator handles your data

In one sentence

Orka Translator captures the meeting audio playing on your own machine, sends short audio segments to a single external service (the OpenAI API) over an encrypted connection to be transcribed and translated, shows you the result, and then forgets it. It stores nothing, contacts no other service, and never touches your microphone unless you are speaking in the meeting yourself.

How a single spoken sentence travels through the system

  Other participants speak in Teams
            ↓  (audio you already hear on your speakers)
  [1] Local capture        Windows system-audio loopback → PCM audio in the app
            ↓  ws://localhost  (stays on your machine)
  [2] Local relay          Embedded Node server running inside the app
            ↓  wss://api.openai.com   TLS-encrypted
  [3] OpenAI Realtime API  transcribe (Whisper) + translate + synthesize speech
            ↓  wss://  TLS-encrypted
  [2] Local relay          forwards result back to the app
            ↓  ws://localhost
  [1] Your screen + speakers   translated text + audio
            ↓
        (nothing is written to disk — the turn exists only in memory)

What leaves your device — stated plainly

For transparency, here is the one and only thing that leaves your computer: the meeting audio you choose to translate is streamed to the OpenAI API, and the transcript and translation (text and synthesized audio) come back. We are deliberately explicit about this because it is the part a security review must assess.

  • Destination: api.openai.com only. No analytics, telemetry, ad networks, or any other third party is contacted.
  • What is sent: the captured meeting audio, plus the fixed translation instructions. Nothing about your identity, account, machine, or files is sent.
  • Encryption in transit: the connection to OpenAI uses TLS (HTTPS/WSS). Audio and text are never sent in clear text over the public internet.
  • Microphone: in listen-only mode the app captures system audio output (what you hear), not a direct microphone tap. Your own voice is carried by Teams as normal and translated by other people's apps, not yours.

What never leaves your device

  • No stored transcripts. Conversations are not written to any database, file, or log. There is no database or cache wired into this build — each translated turn lives in memory only and is discarded when you stop or close the app.
  • No content logging. The local server logs only connection events and an anonymous random session id. Spoken content, transcripts, and translations are never written to the logs.
  • Your API key stays local. The key lives in a local .env file, is read only by the local server, and is used solely as the credential to OpenAI. It is never exposed to the user interface or sent anywhere except OpenAI's authentication header.
  • No background phone-home. The app does not auto-update, beacon, or transmit anything when idle. It is silent unless you press Start Listening.

Processing & retention at OpenAI

Translation is performed by the OpenAI API. Under OpenAI's API data-usage policy, data sent through the API is not used to train OpenAI models, and is retained only transiently for abuse monitoring before deletion. The exact retention window and any Zero-Data-Retention terms depend on the organisation's OpenAI agreement and should be confirmed by the security team against that contract before production use.

Known security considerations (full transparency for review)

Rather than claim the system is flawless, we list the open items so they can be assessed honestly. None of these expose stored data — they are exposure-surface and hardening items.

  • Local server bind address. The embedded server currently binds to all network interfaces (0.0.0.0:3001), so it is reachable from the local network while running. Planned mitigation: bind to 127.0.0.1 (loopback only) so it is reachable solely from your own machine.
  • API key handling. No secret is embedded in the distributed build — it ships with an empty .env. Each user receives their own key, issued and revocable centrally (a per-user subscription/seat model), so usage is attributable per person and there is no shared secret to leak. The key is stored in a local plaintext .env on the user's machine. Planned hardening: OS-level secret storage / Entra-issued credentials instead of a plaintext file.
  • Unsigned build. The POC executable is not yet code-signed, which is why Windows SmartScreen warns on first launch. Planned: Nokia code-signing certificate for distributed builds.
  • Cloud dependency. Translation requires the external OpenAI service; the system is not air-gapped. An on-premise or Nokia-hosted model is a possible future direction if data-residency requires it.

Where this is heading — security as a built-in feature

The architecture above is deliberately simple and inspectable so that controls can be layered on with confidence. The roadmap treats security as a product feature, not an afterthought:

  • Policy-aware routing. Classify content before it leaves the device and route or redact according to Nokia data-handling policy (e.g. block or mask confidential terms automatically).
  • Prescriptive AI guardrails. Use the model itself to flag, in real time, where a conversation may touch sensitive or regulated material — surfacing “this segment looks like it could contain restricted information” and suggesting the safer handling, so risk is caught proactively rather than after the fact.
  • Configurable data boundaries. Per-deployment switches for retention, region/data-residency, and on-premise model hosting, so each organisation can dial the data boundary to its own compliance bar.
  • Audit & consent surfaces. Optional, opt-in audit trails and clear in-meeting consent indicators so participants always know translation is active.

The roadmap items above describe intended future direction and are not yet implemented in this POC build. The "in one sentence", data-flow, "what leaves / never leaves", and "known considerations" sections describe the system as it actually behaves today.

Dev Notes

Multi-party rewrite — 2026-06-11

The app was redesigned from a 1-to-1 "I speak / They speak" pair into a multi-party listen-only translator. Each participant runs the app on their own laptop, picks the language they want to hear, and the app translates everything coming through Teams audio into that language. If a speaker is already speaking the listener's language, no translation is produced.

Key architecture changes

  1. One Realtime session per client, not two. The "outgoing" mic path is removed entirely — your own voice is translated by everyone else's app.
  2. Whisper auto-detects language per turn. The session prompt tells the model to translate to a single target and to stay silent when input is already in target.
  3. Single chronological transcript pane replaces the two-column "What I Say / What They Say" layout.
  4. Feedback-loop suppression in useAudioCapture: system-audio chunks are dropped while AudioPlayer is actively playing translated output. Breaks the Windows render-loopback → capture → retranslate echo.

Known limitations

  • Audio mixing is naive — user hears the original Teams audio AND the translated overlay simultaneously. Use Teams' captions-only or reduce Teams volume during demos.
  • Same-language detection is fully delegated to the model. If it doesn't follow the "stay silent" instruction perfectly you may occasionally get a redundant English-to-English passthrough.
  • The "turn_skipped" event is inferred from response.done producing no audio output — works in practice but not provably airtight.

Earlier Windows packaging fixes (still relevant for next builds)

  • Server's runtime node_modules must be bundled into resources/server/ by electron-builder.
  • @orka/shared's main field must point at the built file, with explicit named re-exports (TS's __exportStar isn't statically detectable by Node's CJS lexer).
  • waitForServer retry budget bumped from 30 to 120 for cold-start.
  • .env shipped without NODE_ENV=development; the production default in main.js wins.
Setup Guide — Nokia Personnel

What this app does

Each participant in a Teams call runs this app locally and picks the language they want to hear in. The app captures meeting audio and translates anything spoken in another language to your chosen language — both as on-screen text and as audio you hear through your speakers/headphones. If a speaker is already speaking your chosen language, no translation is produced.

You only configure your language. Everyone else's translator handles their end.

Step 1 — Copy the zip somewhere sensible

  1. Save Nokia-Orka-Translator-v1.0.1-Portable.zip to Documents or anywhere with write access.
  2. Right-click the zip → Extract All… → pick a destination.
  3. Open the extracted folder. You should see Nokia Orka Translator.exe next to a resources\ folder and a .env file.

No admin rights needed. The portable build runs from wherever you extract it.

Step 2 — Your access key (your "seat")

This tool runs on a per-user subscription model. Each person gets their own access key, issued by Prateek — think of it as your seat. Usage and billing are monitored centrally, so there is no shared key floating around.

  1. Ping Prateek to request a key. You'll receive a value that looks like sk-….
  2. In the extracted folder, open .env in Notepad (it sits next to Nokia Orka Translator.exe).
  3. Paste your key after OPENAI_API_KEY= (no spaces, no quotes). Leave PORT and NODE_ENV as they are.
  4. Save and close. Done — your key stays on your machine and is never shown in the app.

The shipped .env comes empty on purpose — no secret is embedded in the download. If you leave it empty the app will report Incorrect API key on first translation.

Step 3 — First launch

Double-click Nokia Orka Translator.exe. Windows shows a few one-time warnings:

3a. SmartScreen — "Windows protected your PC"

  1. Click More info (small link).
  2. The dialog expands. Check the publisher reads Nokia MS - ORKA team.
  3. Click Run anyway.

3b. Firewall — "Allow access"

  1. Tick Private networks.
  2. Click Allow access.

3c. Microphone permission

When Windows asks "Let Nokia Orka Translator access your microphone?" — click Yes. The mic permission is needed so the screen-capture API can also surface system audio.

3d. Screen-share pop-up

On first capture, Windows briefly shows a screen-sharing picker. It auto-selects the desktop — just wait, it disappears on its own.

Step 4 — Use it in a meeting

  1. Join your Teams call (or play any foreign-language YouTube video to test).
  2. Pick your language in the "I want to hear in" dropdown.
  3. Click Start Listening.
  4. The transcript fills as people speak: italic original, then the translation in your language. Same-language turns show "already in your language — no translation needed" and stay silent.
  5. Click Stop when done.

Headphones give the cleanest experience. Speaker playback works but can echo through Teams audio.

Step 5 — When you're done

  • Closing the window hides to the system tray (Nokia-blue square icon).
  • Right-click the tray icon → Quit to fully exit.
  • To uninstall: delete the extracted folder. That's it.

Notes for Nokia laptops & customer-protected systems

This is an unsigned proof-of-concept that talks to an external service (OpenAI) over the internet. On managed devices that matters — please read before running:

  • No admin rights needed. The portable build runs from wherever you extract it. It does not install drivers, services, or registry entries.
  • SmartScreen warning is expected. Because the build isn't code-signed yet, Windows shows "Windows protected your PC." Click More info → Run anyway (see Step 3a). This will go away once we ship a Nokia-signed build.
  • Endpoint protection (Defender / CrowdStrike / etc.) may quarantine it. Managed Nokia endpoints can block unsigned executables. If the exe vanishes after extraction or won't launch, ask IT for a folder exclusion for the extracted folder, or run it from an IT-approved location. Do not disable your security tooling.
  • Outbound internet to api.openai.com (HTTPS / port 443) is required. Everything else stays on your machine, but translation needs this one outbound connection. On networks behind a corporate proxy, that host must be reachable — if translation hangs or errors on a proxied/VPN network, IT may need to allow api.openai.com.
  • Firewall prompt on first launch is for the app's own local server (localhost). Tick Private networks and allow — it is not opening you up to the internet.
  • Customer-protected / customer-managed systems: please do not run this on a customer's device or network without that customer's explicit approval. The tool sends meeting audio to OpenAI for translation (see the Architecture & Security panel for exactly what leaves the machine). On customer-protected systems that egress may breach the customer's data policy. Test on your Nokia laptop, not on customer-protected systems, unless the customer has signed off.
  • What is sent off-device: only the meeting audio you choose to translate, to OpenAI, over TLS. No transcripts are stored or logged. Full details are in the Architecture & Security panel — share that with your security contact if asked.

Troubleshooting

  • Blank window for ~10 sec — first launch starts the embedded server. Wait.
  • TRANSLATION_ERROR: Incorrect API key — .env is missing or empty. Check Step 2, fully relaunch.
  • "Cannot find server" — resources\server\ folder is corrupted. Re-extract the zip.
  • Port 3001 already in use — edit .env, change to e.g. PORT=3055, relaunch.
  • Garbled / repeating translations — speaker output is bleeding into system-audio capture. Use headphones, or lower Teams volume.
  • Antivirus quarantines the exe — unsigned POC build. Add a folder exclusion or ping Prateek.
